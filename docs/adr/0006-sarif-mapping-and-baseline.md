# ADR-0006 — SARIF 2.1.0 mapping, baseline format, and suppressions

- **Status:** Accepted
- **Date:** 2026-06-10
- **Deciders:** ActaClad founding team
- **Supersedes:** none

> ADRs are immutable once Accepted. To change a decision, write a new ADR that
> supersedes this one. Number ADRs sequentially. Keep them short.

---

## Context

SARIF 2.1.0 output is mandatory and must validate against the official schema
(ADR-0001 D7). `backlog.md` lists the baseline/suppression format as an open
question that must be resolved before M2. Plumbline's model (5 severities ×
3 confidences × 4 pillars) is richer than SARIF's 4 levels, so the mapping
must be lossless via `properties`.

## Decision

### D1 — SARIF structure

One `run` per scan. `tool.driver.name = "Plumbline"`,
`informationUri = https://github.com/actaclad/plumbline`, `semanticVersion` =
package version. Every loaded rule (not just firing ones) appears in
`tool.driver.rules` with: `id`, `name` (slug of the title),
`shortDescription` (title), `fullDescription` (why_it_matters),
`help.text`/`help.markdown` (remediation), `helpUri` (anchor into the
published rule catalog), `defaultConfiguration.level` (mapping below), and
`properties: { "plumbline/pillar", "plumbline/severity",
"plumbline/confidence", "tags": [standards…, pillar] }`. For SEC rules,
`properties."security-severity"` is set so GitHub code scanning ranks them.

### D2 — Severity → SARIF level mapping

| Plumbline | SARIF `level` |
|---|---|
| Blocker, Critical | `error` |
| Major | `warning` |
| Minor, Info | `note` |

The full (severity, confidence) pair always rides in `result.properties`, so
no information is lost. Confidence additionally maps to `result.rank`
(High=90, Medium=50, Low=10) for viewers that sort by rank.

### D3 — Results, locations, fingerprints

Each finding → one `result` with `ruleId`, `message.text` (the instance
message; `why_it_matters` is on the rule), one `physicalLocation` with
`artifactLocation.uri` (POSIX-relative) + `uriBaseId: "SRCROOT"`, and a
`region` (`startLine`, `startColumn` = internal column + 1 — SARIF is
1-based, `endLine` when known), plus `partialFingerprints:
{ "plumblineFingerprint/v1": <fingerprint> }` (ADR-0002). Taint-rule witness
paths are emitted as `codeFlows` when present. `columnKind =
"unicodeCodePoints"`. Analyzer errors (crashed detector/adapter) are emitted
as `invocations[0].toolExecutionNotifications` with `level: error`, and
`executionSuccessful` stays `true` (the scan completed; the gate, not SARIF,
decides pass/fail). No timestamps are emitted — byte-reproducibility
(ADR-0002 D3) outranks optional metadata.

### D4 — Schema validation is a test, with a vendored schema

The official SARIF 2.1.0 JSON schema is vendored under `tests/data/` and
every SARIF test validates output against it with `jsonschema`. No network in
tests. License/provenance of the schema file is recorded alongside it.

### D5 — Baseline file: Plumbline-owned minimal JSON

`.plumbline-baseline.json`, generated by `plumb baseline` after a scan:

```json
{
  "version": 1,
  "algorithm": "v1",
  "findings": [
    { "fingerprint": "a1b2c3d4e5f60718",
      "rule_id": "PLB-RES-001",
      "file": "app/agent.py" }
  ]
}
```

Matching is by fingerprint only; `rule_id`/`file` are human-readable context
(and let `plumb baseline --prune` explain removals). Baselined findings are
still reported but marked suppressed, are excluded from the Quality Gate and
from scoring, and appear in SARIF as `suppressions: [{ "kind": "external" }]`.
Rejected: using a previous SARIF log as the baseline (SARIF logs are large,
order-sensitive to diff, and overkill for "the set of accepted
fingerprints"); rejected: GitLab-style per-line hashes (ADR-0002 already
solved stability).

### D6 — Inline suppression comment

`# plumb: ignore[PLB-XXX-NNN]` on the finding's line suppresses that rule on
that line; a reason may follow after `--`. Multiple IDs are comma-separated.
A bare `# plumb: ignore` (no rule ID) is **invalid** and reported as an
analyzer warning — blanket suppression hides defects silently. Inline
suppressions appear in SARIF as `suppressions: [{ "kind": "inSource" }]`.
Comments are read via `tokenize` (the AST layer drops comments — ADR-0009).

## Consequences

- `plumb scan --sarif out.sarif` uploads cleanly to GitHub code scanning;
  fingerprint stability prevents alert churn.
- Teams adopt incrementally: scan → `plumb baseline` → gate only on new
  findings from day one.
- The vendored schema pins what "valid SARIF" means for tests; bumping it is
  a deliberate, reviewed change.
- Suppression semantics (excluded from gate + scoring, visible in reports) are
  uniform across baseline and inline mechanisms.
